# Engineering Package Output Contract

## Purpose

This output contract defines the required structure for a Daedalus engineering package.

An engineering package is the main deliverable Daedalus produces after an engineering request has been reviewed and accepted.

## Rules

Daedalus must produce engineering packages as proposed work unless a human explicitly marks the package as approved.

Daedalus must not claim implementation has occurred.

Daedalus must include human approval gates for infrastructure-impacting work.

Daedalus must include validation and rollback sections when the work could affect infrastructure, access, backups, or security boundaries.

Daedalus must clearly separate assumptions from confirmed facts.

## Required Output Structure

Every engineering package must include the following sections:

1. Package metadata
2. Request summary
3. Scope
4. Current state
5. Desired outcome
6. Assumptions
7. Risks
8. Proposed engineering plan
9. Generated artifacts
10. Validation checklist
11. Rollback plan
12. Threat model notes
13. Human approval gate
14. Completion criteria
15. Follow-up actions

## Required Markdown Format

```markdown
# Engineering Package: <Title>

## Package Metadata

**Package ID:**  
**Date:**  
**Author:** Daedalus  
**Status:** Proposed | Approved | Rejected | Superseded  
**Related Request:**  
**Related ADRs:**  
**Related Memory Records:**  
**Risk Level:** Low | Medium | High | Critical  

## Request Summary

Summarize the original engineering request.

## Scope

### In Scope

List what this package covers.

### Out of Scope

List what this package does not cover.

## Current State

Describe the current known state.

Separate confirmed facts from assumptions.

## Desired Outcome

Describe the target end state.

## Assumptions

List assumptions used while creating this package.

## Risks

List technical, operational, security, and rollback risks.

## Proposed Engineering Plan

Provide the implementation approach in ordered phases.

Do not include destructive actions without explicit warnings and rollback notes.

## Generated Artifacts

List generated files, configs, commands, IaC, documentation, or templates.

Use subsections for each artifact.

## Validation Checklist

Provide clear validation steps.

Each check should include the expected result.

## Rollback Plan

Describe how to safely reverse the proposed change.

If rollback is not applicable, explain why.

## Threat Model Notes

Identify trust boundaries, likely threats, and mitigations.

## Human Approval Gate

State exactly what requires human approval before use.

## Completion Criteria

List what must be true for this package to be considered complete.

## Follow-Up Actions

List recommended next actions.
```

## Status Definitions

### Proposed

Generated by Daedalus but not approved for use.

### Approved

Reviewed and approved by a human.

### Rejected

Reviewed and rejected by a human.

### Superseded

Replaced by a newer package or decision.

## Quality Requirements

A valid engineering package must be:

- Clear enough for another engineer to review
- Explicit about assumptions
- Explicit about risks
- Reversible when infrastructure-impacting
- Validatable through checklist steps
- Bound to a human approval gate
- Traceable to the original request
